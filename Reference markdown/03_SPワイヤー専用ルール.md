# 03_SPワイヤー専用ルール

## 0. このファイルの目的

このファイルは、PCワイヤーからSPワイヤーへ変換する際に、

- 誤タップ
- 情報過多
- CSS/HTML干渉
- WordPress実装時の手戻り
- Gutenberg編集による崩れ
- iPhone系ブラウザでの表示不具合

を減らすためのルールを定義する。

このファイルは、**見た目をSPっぽくするためのメモではなく、SPワイヤーをそのままWordPress実装へ渡しやすくするための設計ルール**である。

---

## 1. 基本原則

### 1-1. SPワイヤーは「PCの縮小版」ではない

SPワイヤーは、PCの見た目をそのまま縮小するものではない。
SPでは以下を優先する。

1. 主要メッセージの即時理解
2. 主要CTAへの到達しやすさ
3. 読みやすさ
4. 誤タップ防止
5. 実装の単純性
6. 更新運用のしやすさ

したがって、PCで横に並んでいる情報でも、SPでは縦積み・省略・折りたたみ・順序変更を許可する。

### 1-2. SPの判断優先順位

SP化の判断は、以下の優先順位で行う。

1. **機能要件**: CTA、フォーム、導線、ナビが成立するか
2. **アクセシビリティ**: タップ、可読性、見出し、ラベル、コントラスト
3. **実装安定性**: CSS/HTMLが単純で、WordPressで分割しやすいか
4. **運用性**: 編集者が壊しにくいか
5. **視覚的一致**: PCと似ているか

PCと完全一致させることより、SPで壊れず使いやすいことを優先する。

### 1-3. 1セクション1目的を原則とする

各セクションは、原則として1つの役割だけ持たせる。

- ヒーロー
- サービス説明
- 実績
- 比較
- FAQ
- CTA
- フォーム

複数の役割を1セクションに混ぜると、SPで縦積みした際に順序が破綻しやすく、`get_template_part()` による分割もしにくくなる。

## 2. ファクトに基づく最低基準

### 2-1. タップターゲット

- WCAG 2.2 Level AA の最低基準は **24 × 24 CSS px**
- Appleはボタンのヒット領域として **44 × 44 pt以上** を一般ルールとしている
- Android/Material系では **48 × 48dp程度**、およそ **7〜10mm** が推奨されている

このProjectでは、**主要なボタン・メニュー・アイコンボタンは 44 × 44px 以上**を採用する。

補足:
- 24pxは「最低基準」に近い
- 実務では44px以上の方が事故が少ない
- 隣接ターゲット同士は最低8px以上離す

### 2-2. モバイルナビゲーション

通常のサイトナビゲーションでは、ARIAの`menu`ロールを安易に使わない。
WAI-ARIA Authoring Practicesでは、一般的なサイトナビには複雑な`menu/menubar`ではなく、**Disclosureパターン**が適しているとされる。

このProjectでは、ハンバーガーメニューは以下を原則とする。

- 開閉トリガーは`button`
- `aria-expanded`を持つ
- `aria-controls`で開閉対象を指す
- Escapeで閉じられる
- PC/SPでHTMLを二重化せず、可能な限り同一構造を使い、CSSとJSで表示制御する

### 2-3. 画像の比率維持

SPで画像が崩れる主因は、`width: 100%` と固定`height`の併用である。
画像のトリミング前提の領域は、`aspect-ratio` と `object-fit: cover` を使う。

注意:
- `aspect-ratio` は、幅か高さのどちらかがautoでないと効かない
- 画像全体を見せたい場合は`cover`ではなく`contain`を検討する

### 2-4. ビューポート高さ

モバイルブラウザでは、従来の`vh`は実質的にlarge viewport相当として扱われることがあり、アドレスバー表示時にコンテンツが隠れることがある。
`dvh`は動的UIに追従できる一方、スクロール中に再計算されるため、レイアウトが伸縮することがある。

このProjectでは、**本文系レイアウトを`100vh`前提で組まない**。
全画面演出が必要な箇所だけ、用途に応じて`svh`/`dvh`を使う。

### 2-5. WordPress編集UIの制御

WordPressでは、`theme.json`で色・タイポグラフィ・余白などのUIを制御できる。
また、Block Locking APIでブロックの移動・削除・編集制限を設定できる。

このProjectでは、実装前提として以下を採用する。

- 色・タイポ・余白は`theme.json`で固定寄りに管理する
- レイアウト崩れの原因になる自由入力は必要最小限にする
- 重要セクションはBlock Lockingの対象候補として扱う

## 3. SPワイヤー化ルール

### 3-1. セクション順序ルール

SPでは、各ページのセクション順序を以下の原則で整理する。

1. ページの結論
2. 主要CTA
3. 補強材料（実績・理由・比較・事例）
4. 詳細説明
5. FAQ
6. 最終CTA
7. フォーム

PCで視線誘導のために左右配置していた要素は、SPではこの順序に並べ替えてよい。

### 3-2. 横並び要素の縦積みルール

PCで横並びの要素は、SPでは以下の優先順位で処理する。

1. **1カラム化**
2. 重要順に縦積み
3. 長文は折りたたみ
4. 補助情報は下段へ移動

対象:
- 3カラムカード
- 比較表
- 実績一覧
- スタッフ紹介
- FAQ導入部
- CTA横並び

原則として、SPで2カラムを維持するのは以下だけに限定する。

- 数字が短い実績カード
- 小さなアイコン付きの要約カード
- 画像サムネイル一覧

### 3-3. ファーストビューの変換ルール

ヒーローは以下の順で縦積みする。

1. ラベルまたは補助文
2. h1
3. 説明文
4. 主CTA
5. 副CTA
6. 補足情報
7. ビジュアル

禁止:
- SPでも画像と文章を左右並列で維持する
- h1の途中改行を見た目固定で埋め込む
- CTAを画像の下深くに追いやる

### 3-4. 比較表・料金表の変換ルール

比較表や料金表は、SPで以下のどちらかに変換する。

- **カード型へ変換**
- **横スクロール前提のテーブルにする**

優先はカード型。
ただし、列比較そのものが価値である場合だけ横スクロールを許容する。

禁止:
- 3列以上の比較表を縮小してそのまま入れる
- セル内テキストが改行だらけになるまま放置する

### 3-5. 動的コンテンツのループ単位を明示する

ブログ、お客様の声、事例、FAQなどは、ワイヤー段階で**1件分の単位**を定義しておく。

例:
- `.c-post-card`
- `.c-voice-card`
- `.c-faq__item`

「後でループにすればよい」と考えて静的レイアウトで作るのは禁止。
WordPress側で構造を作り直すことになり、SP崩れの原因になる。

### 3-6. フォームのSPルール

SPでは、フォームは原則として1カラム。

- ラベルは見える形で置く
- 必須表記は視認できる形で付ける
- 項目幅は100%
- 送信ボタンは44px以上
- エラー文は入力欄の近くに出せる前提で設計する

禁止:
- プレースホルダーだけで項目名を表現する
- 氏名/電話番号などを無理に横並び化する
- 送信ボタンを画面端に寄せて小さくする

## 4. SPデザイントークンの標準値

以下は厳密な標準規格ではなく、このProjectで採用する**運用基準値**である。
目安を固定して、ページごとのブレを防ぐことが目的。

### 4-1. ブレークポイント

```css
:root {
  --bp-sp: 768px;
  --bp-tab: 1024px;
}
```

ルール:
- ブレークポイントはファイルごとにバラバラにしない
- `767px` `769px` `800px`などを混在させない
- すべてグローバルトークン経由で使う

### 4-2. コンテナ余白

標準:
- SP左右余白: **16px**
- 狭い画面で窮屈なら **20px** を許容
- セクション上下余白: **48px / 56px / 64px** の3段階を基本とする

### 4-3. フォント基準

標準:
- 本文: **16px / line-height 1.7**
- 補足文: **14px / line-height 1.6**
- セクション見出し: **24〜28px**
- ヒーローh1: **28〜32px**
- ボタン文字: **16px以上**

備考:
- 14px本文を常用しない
- 小さく見せたい場合は色や字幅で調整し、本文自体は読みやすさ優先
- 見出しや余白に`clamp()`を使うのは可

### 4-4. 余白スケール

推奨スケール:
- 4
- 8
- 12
- 16
- 24
- 32
- 40
- 48
- 64

ルール:
- 端数の多発を避ける
- 18, 22, 26のような値を濫用しない
- gap / margin / paddingはこのスケールに寄せる

### 4-5. ボタン基準

標準:
- 高さ: **44px以上**
- 左右padding: **16px以上**
- 主要CTA幅: 原則100% または十分な横幅
- アイコンボタン: **44 × 44px以上**

## 5. HTML / CSS / JS 命名規則

### 5-1. 命名原則

このProjectでは、**プロジェクト固有プレフィックス + BEM** を原則とする。

例:

```css
.mf-hero {}
.mf-hero__title {}
.mf-hero__actions {}
.mf-hero--compact {}
.is-open {}
.js-nav-toggle {}
```

### 5-2. 使い分け

- `mf-` : プロジェクト固有ブロック
- `__` : 要素
- `--` : バリエーション
- `is-` : 状態
- `js-` : JavaScriptフック専用

### 5-3. 禁止クラス

以下のような汎用クラスは、衝突しやすいので原則禁止。

- `.title`
- `.text`
- `.inner`
- `.wrap`
- `.item`
- `.btn`
- `.active`
- `.open`

理由:
- プラグインCSS
- 既存テーマCSS
- Gutenberg生成クラス

と衝突しやすい。

### 5-4. JSフックと見た目クラスを分離する

例:

```html
<button class="mf-header__toggle js-nav-toggle" ...>
```

ルール:
- `js-`クラスにはCSSを当てない
- 見た目変更は`is-open`など状態クラスで行う
- JSで`.title`や`.inner`を拾わない

### 5-5. ワイヤー専用クラスの扱い

ワイヤー専用プレースホルダーは、本番に残さない。

例:
- `.ph-img`
- `.ph-logo`
- `.wf-note`

ルール:
- SPワイヤーでは使用可
- 実装前に削除対象として明記
- 本番CSSの依存先にしない

## 6. WordPress実装を見越した構造ルール

### 6-1. セクションはフラットに並べる

`get_template_part()`で切り出しやすくするため、`<main>`直下の各セクションは原則フラットに並べる。

推奨:

```html
<main>
  <section class="mf-hero">...</section>
  <section class="mf-proof">...</section>
  <section class="mf-service">...</section>
  <section class="mf-faq">...</section>
  <section class="mf-cta">...</section>
</main>
```

非推奨:
- 大きなwrapperの中に複数セクションを曖昧に入れる
- 1ファイルで複数セクションの責務を持たせる

### 6-2. パーツ名とファイル名を先に決める

例:

```text
parts/section/hero.php
parts/section/proof.php
parts/section/service.php
parts/section/faq.php
parts/section/cta.php
```

ルール:
- ワイヤー段階で「どこで切るか」を明記
- 1セクション1パーツを原則
- 共通CTAや共通見出しは別パーツ候補として扱う

### 6-3. 動的領域は固定ページと混ぜない

以下はカスタム投稿またはループ想定で設計する。

- ブログ
- お客様の声
- 事例
- FAQ（件数増減がある場合）
- お知らせ

静的ページの途中に、静的HTMLで大量に埋め込む設計は避ける。

### 6-4. theme.json前提でワイヤーを作る

以下の可変項目は、なるべく`theme.json`側で固定できる前提で考える。

- 色
- フォントサイズ
- 余白
- ブロック間隔

ワイヤー側で細かい例外を増やすほど、実装側で`theme.json`の統制が効きにくくなる。

### 6-5. 編集者が触ってよい領域を限定して設計する

ワイヤー時点で、以下を区別する。

- 編集可
- 原則固定
- CMSループ管理
- 管理画面から画像だけ差し替え

この区別がないと、実装後に自由編集領域が増え、SP崩れが起こりやすい。

## 7. Pitfall一覧

### 7-1. PCとSPでHTMLを二重管理する

問題:
- PC用DOM
- SP用DOM

を別々に持つと、修正漏れ・SEO重複・表示崩れが起きやすい。

原則:
- 可能な限り同一HTML
- レイアウト差分はCSSで吸収
- 本当に別構造が必要な場合だけ限定的に分岐

### 7-2. 固定heightでテキスト領域を制御する

問題:
- テキスト増加
- 翻訳差分
- エラーメッセージ表示

で簡単に崩れる。

原則:
- テキスト領域に固定heightを使わない
- 必要なら`min-height`
- 高さ固定するのは画像ラッパーや演出領域に限定

### 7-3. ワイヤーCSSをそのまま本番に残す

問題:
- 仮スタイルの残存
- クラス衝突
- 予期しない上書き

原則:
- wireframe-base.cssは本番へ持ち込まない
- 本番CSSは別管理
- クラス名だけ整合を取る

### 7-4. 画像の見た目合わせをインラインstyleで済ませる

問題:
- 再利用不可
- breakpoint対応困難
- Gutenberg内で再編集されにくい

原則:
- インラインstyle禁止
- 高さや余白はクラス/トークンで管理

### 7-5. ハンバーガーを`div`や`a`で実装する

問題:
- キーボード操作
- スクリーンリーダー通知
- 状態管理

が弱くなる。

原則:
- トリガーは`button`
- `aria-expanded` / `aria-controls`
- 開閉状態は属性と見た目を同期

### 7-6. FAQを見た目だけで作る

問題:
- 構造化データと対応しにくい
- 折りたたみJSも後付けになりやすい

原則:
- 1質問1回答の単位を最初から持たせる
- FAQパーツをループ単位で設計する

### 7-7. 余白と文字サイズがページごとに散る

問題:
- サイト全体の統一感低下
- CSS肥大化
- Gutenberg設定とも衝突

原則:
- トークン化
- 余白と文字サイズは段階化
- 例外は最小限

### 7-8. sticky / fixed要素を後付けする

問題:
- 画面占有
- キーボードフォーカス干渉
- iPhone Safariでの視認性問題

原則:
- SPワイヤー段階で位置と高さを決める
- CTA固定は主CTAだけに限定
- stickyヘッダーとfixed CTAの併用は慎重に扱う

## 8. SPワイヤー作成時のチェックリスト

### 8-1. 操作性

- [ ] 主要CTAは44px以上か
- [ ] 主要ターゲット間に8px以上の余裕があるか
- [ ] ハンバーガーはbutton前提か
- [ ] Escapeで閉じる前提か
- [ ] フォーム送信ボタンは十分大きいか

### 8-2. レイアウト

- [ ] PCの横並びを無理に維持していないか
- [ ] ファーストビューで結論とCTAが先に来るか
- [ ] 比較表/料金表のSP処理方針が決まっているか
- [ ] テキスト領域に固定heightを使っていないか
- [ ] 画像比率の扱いが決まっているか

### 8-3. 実装

- [ ] セクションはフラット構造か
- [ ] `get_template_part()` の切り出し単位が見えているか
- [ ] 動的領域のループ単位が定義されているか
- [ ] ワイヤー専用クラスの削除ルールがあるか
- [ ] PC/SPでHTML二重管理になっていないか

### 8-4. WordPress運用

- [ ] `theme.json`で固定したい項目が整理されているか
- [ ] 編集可/固定/ループ管理の区別があるか
- [ ] ブロックロック対象候補が分かっているか
- [ ] プラグインCSS干渉を起こしそうなクラスがないか

## 9. 実装者への引き継ぎメモに必ず含める項目

SPワイヤー確定時は、以下を別紙または注記で必ず渡す。

1. セクション順序
2. 削除した要素
3. 折りたたみ対象
4. ループ対象
5. 共通化対象
6. fixed/sticky要素の有無
7. theme.jsonで固定したい項目
8. 編集可/固定の区別
9. 禁止実装（インラインstyle、DOM二重化など）

## 10. このファイルの位置づけ

このファイルは、以下の中間に位置する。

- PCワイヤー確定ファイル
- WordPress実装ナレッジファイル

つまり、**PCワイヤーの情報設計**と、**WordPress実装の制約**をつなぐための橋渡しファイルである。

このファイル単体で、

- SPワイヤー作成者
- 実装者
- レビュー担当

が同じ判断基準で会話できる状態を目指す。

## 付録A. 推奨コード断片

### A-1. ハンバーガーボタン

```html
<button
  class="mf-header__toggle js-nav-toggle"
  type="button"
  aria-expanded="false"
  aria-controls="global-nav"
  aria-label="メニューを開く"
>
  <span class="mf-header__toggle-line"></span>
  <span class="mf-header__toggle-line"></span>
  <span class="mf-header__toggle-line"></span>
</button>
```

### A-2. 画像ラッパー

```css
.mf-card__media {
  aspect-ratio: 16 / 9;
  overflow: hidden;
}

.mf-card__media img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}
```

### A-3. clampを使う見出し例

```css
.mf-section-title {
  font-size: clamp(24px, 5vw, 28px);
  line-height: 1.3;
}
```

### A-4. ブレークポイントトークン例

```css
:root {
  --bp-sp: 768px;
  --bp-tab: 1024px;
}

@media (max-width: 768px) {
  .mf-feature-list {
    display: flex;
    flex-direction: column;
    gap: 16px;
  }
}
```
